# Prompt Log: QA Analyst and Documentation Lead

## Entry 1 - (03/30/2026)

### **Task(s)**

-- To generate the needed test cases for the Lost & Found Knowledge Board --

### **Prompt (or Summary) provided to the AI**

-- Requested a structured set of at least 10 test cases for the NEU Lost & Found platform. Emphasized coverage of CRUD operations, search, navigation, edge cases, negative tests, and institutional email validation (@neu.edu.ph). Also requested the inclusion of AI Photo Matching test cases marked as low priority. --

### **Output**

-- The AI generated 14 standard test cases (TC-001 to TC-014) covering Authentication, Item Management (Create, Update, Delete), Navigation, Search, Communication, Admin Moderation, and AI Photo Matching. --

### **What you changed/improved/rejected from the AI's output**

-- Rejected the AI Photo Matching test cases (TC-013, TC-014), removing them completely from the final document. Improved the Authentication steps (TC-AUTH-FUNC-001 and TC-AUTH-NEG-001) by removing the AI's manual password entry steps and replacing them with Google Authentication validation steps. --

### **Reason (Why?)**

-- The actual system uses Google Authentication for institutional emails, so the manual password registration and login steps generated by the AI were structurally incorrect. The AI Photo Matching feature was deemed low priority and excluded from the core test suite to focus strictly on core features. --

### **What you learned or the decision you made as a result**

-- AI establishes a strong structural baseline for test cases, but it assumes standard web authentication (email/password) unless explicitly told about SSO integrations. Human intervention is necessary to align steps with actual architectural decisions. --

## Entry 2 - (03/30/2026)

### **Task(s)**

-- To add more test cases and make the Test Case ID more unique to each feature --

### **Prompt (or Summary) provided to the AI**

-- Instructed the AI to retain all existing test cases but update the Test Case IDs to a mandatory scalable format: TC-[MODULE]-[TYPE]-[NUMBER] (e.g., TC-AUTH-FUNC-001). Also prompted to add new test cases to improve coverage for edge cases, missing inputs, and security boundaries without creating duplicates. --

### **Output**

-- The AI successfully migrated all existing test cases to the new ID format, grouping them by modules (AUTH, POST, CRUD, SRCH, NAV, MSG, ADMIN, AI). It also generated new negative and edge cases, such as TC-POST-EDGE-001 (missing required fields) and TC-CRUD-SEC-001 (unauthorized editing). --

### **What you changed/improved/rejected from the AI's output**

-- Modified TC-CRUD-EDGE-001 (User submits an edit without changes): The AI suggested the system should accept the submission gracefully, but I changed it so the system does not activate the Save Changes button. Rejected TC-AUTH-NEG-002 (unregistered email login). Added TC-AUTH-NEG-003 to cover a blocked user attempting to log in. --

### **Reason (Why?)**

-- From a UI/UX standpoint, a "Save" button should be disabled if no state changes occur to prevent unnecessary API calls. TC-AUTH-NEG-002 was rejected because Google SSO natively handles unregistered email rejection. I manually added the blocked user scenario (TC-AUTH-NEG-003) as it represents a critical internal moderation state the AI missed. --

### **What you learned or the decision you made as a result**

-- AI-generated expected results often default to backend error handling rather than proactive frontend UI constraints. QA documentation must actively enforce user-centric UI behaviors like disabled buttons. --
